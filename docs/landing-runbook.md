# Landing Contributor Runbook

This runbook describes day-to-day editing, validation, and deploy safety for the landing page on `main`.

## 1. Prerequisites

- Python 3.12
- Quarto 1.3.340 (matches CI pin)

## 2. One-time local setup

```bash
python3 -m venv .venv
source .venv/bin/activate
python -m pip install --upgrade pip
python -m pip install -r requirements-dev.txt
```

## 3. What to edit

- Page structure and static text: `_sections/*.qmd` and `index.qmd`
- Current/previous instances and URLs: `data/instances.yml`
- Dynamic label text (CTA/button labels): `data/ui.yml`
- Styling: `styles.css`

Do not edit `_generated/*` manually. It is generated by `scripts/generate_landing.py`.

## 4. Local validation and render sequence

```bash
python -m pytest -q
python scripts/generate_landing.py
quarto render
```

Expected outputs:

- Generated dynamic partials in `_generated/`
- Rendered site in `_site/`

## 5. Validation rules (high-impact)

`data/instances.yml` must satisfy:

- Exactly one entry with `status: current`
- That current entry must be `visible: true`
- `slug` is 4 digits (`0000` allowed, otherwise valid `YYMM` month)
- Relative `instance_url` must match slug (`./{slug}/`)
- `registration_url` may be empty, relative, or `http(s)`, but not `//...`
- `sort_key` must be an integer and not a boolean

`data/ui.yml` must satisfy:

- Only documented keys are allowed
- Provided values must be non-empty strings
- Allowed keys are: `hero_view_current_label`, `hero_registration_open_label`,
  `current_instance_note_label`, `current_instance_open_label`,
  `current_instance_registration_label`, `previous_instances_title`,
  `no_previous_instances_label` (see `data/ui.yml`).

## 6. Deploy safety contract

Landing deploy job must:

- Copy only landing-owned root files (`index.html`, `styles.css`)
- Mirror only landing-owned directories (`site_libs/`, `index_files/`, `img/`) with per-dir delete
- Never run root-level delete sync (`rsync -a --delete _site/ gh-pages/`)
- Keep `.nojekyll` in `gh-pages`
- Remove leaked raw config (`gh-pages/data`, root YAML files)
- Landing and instance deploy workflows share `deploy-gh-pages` concurrency control
- `gh-pages` is machine-managed output; manual commits are exceptional recovery only

## 7. Recovery steps if deploy assertions fail

1. Fix the workflow logic in `main` and re-run deploy.
2. If a bad artifact already exists in `gh-pages`, remove it with a targeted cleanup commit on `gh-pages`.
3. Re-run deploy from `main` to republish expected landing artifacts.

Do not add a permanent assertion bypass in workflows.
